# 全库入库规则

> 用于规范整个知识库的 Agent 写入、整理、迁移和扩充动作

---

## 一、总原则

- 先判断位置，再写内容
- 先判断主归属，再补双向链接
- 先去风险，再做润色
- 先保长期可维护，再保短期可用

---

## 二、模块归属规则

| 信息类型 | 应进入位置 |
|------|------|
| 概念定义、传统背景、学科解释 | `01-核心概念` |
| 产品、渠道、内容、转化、运营打法 | `02-商业运营` |
| 课纲、讲义、直播讲稿、作业、题库 | `03-教育内容` |
| 平台边界、风险表达、法律伦理 | `04-合规规范` |
| 公司主体、项目计划、执行排期 | `05-项目资料` |
| 角色、岗位、提示词、协作分工 | `06-角色设定` |
| 经典阅读、原始文本、导读索引 | `07-经典书籍` |
| Agent 接入、规则、模板、命名 | `08-LLM-Wiki知识管理` |

---

## 三、入库前检查

每次新增或重构文档前，至少做四件事：

1. 搜索仓库里是否已有同主题文档
2. 判断这次是补充、迁移还是重写
3. 判断是否涉及合规风险
4. 判断是否需要同步更新总索引或模块导航

---

## 四、文档处理规则

### 适合直接新增的情况

- 仓库内没有同类文档
- 新主题已经能明确归档
- 有明确上下文和落点

### 适合补充原文档的情况

- 现有文档结构稳定，只是内容偏少
- 本次新增内容与原文主题高度一致

### 适合拆分的情况

- 一篇文档承载多个主题
- 同时混有知识、营销、合规、课程内容
- 单篇已经不利于 Agent 检索和后续维护

---

## 五、链接规则

每次写回后，至少检查是否需要链接到：

- 模块根 README
- 同主题主文档
- 相关项目文档
- 相关合规条目

避免只写孤立文档，不挂入口。

---

## 六、版本规则

- 主版本只能有一个
- 历史版本保留时必须说明用途
- 不使用“最终版”“新版”“修改版2”这类模糊命名

详细见：[[08-LLM-Wiki知识管理/命名与版本规范]]

---

## 七、禁止直接入库的内容

- 无来源的碎片结论
- 承诺结果的营销话术
- 未去风险的课程原始口播稿
- 医疗、法律、财税类高风险判断
- 与项目当前定位冲突的旧表达

---

## 八、回流原则

- 课程使用中出现的高频问题，回流到课程模块或核心概念
- 营销使用中验证有效的表达，回流到商业运营
- 发现的风险表达，回流到合规规范
- 发现的结构性问题，回流到本模块
